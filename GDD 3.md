## 1. 游戏概述 (Overview)

### 1.1 核心概念

《浮岛浮光》是一款等距立体（2.5D Isometric视角）的逆向跑酷、动态路径规划休闲策略小游戏。传统跑酷中玩家控制角色躲避障碍；本作中，角色（发光水晶块）沿固定逻辑自动前进，玩家扮演“世界构建者”，在水晶块滑入虚空前，实时利用手牌（盲盒刷新的晶体道路片段）拼接、修正、覆盖前方的道路，引导其安全抵达终点或无限延伸。

### 1.2 核心体验与心流 (Core Flow)

- 低门槛：单指拖拽放置，3秒上手。
    
- 高频微操：盲盒手牌的随机性逼迫玩家在“子弹时间”与“极限救场”中进行瞬时动态规划。
    
- 高级感：没有冗余的 Q 版原画，纯靠莫兰迪色彩渐变与代码动效营造高雅的解谜氛围。
    

## 2. 核心玩法与机制 (Mechanics)

### 2.1 游戏基本舞台：双层网格系统

游戏虽然呈现 2.5D 视觉，但底层是纯粹的 2D 矩阵逻辑网格（默认 $10 \times 10$）。

- 逻辑坐标：以 $(row, col)$ 表示，$(0,0)$ 为左下角。
    
- 空间转换：Cocos 中使用公式将逻辑网格坐标转换为 2.5D 屏幕坐标：  
    $$screenX = (col - row) \times (tileWidth / 2)$$  
    $$screenY = (col + row) \times (tileHeight / 2)$$
    

### 2.2 实体属性与状态机

#### A. 跑酷质点（Runner - 水晶块）

- 状态：IDLE（起点蓄力）、RUNNING（滑行中）、BULLET_TIME（时空减速）、DEAD（坠落）。
    
- 数据属性：当前网格坐标 $(r, c)$、当前滑行进入方向（up|down|left|right）、当前颜色属性（如无色、红色、蓝色）。
    

#### B. 晶体道路片段（Tile - 手牌/网格）

每个晶体块的核心是一个转向映射表（Direction Mapping Matrix）。

  
  
  

TypeScript

interface TileConfig {  
    id: string;          // 组件唯一标识  
    type: string;        // 'straight' | 'curve' | 'cross'  
    // 转向核心逻辑：[进入方向] -> [流出方向]  
    routing: {  
        from_left?:  'right' | 'up' | 'down' | 'dead';  
        from_right?: 'left' | 'up' | 'down' | 'dead';  
        from_up?:    'down' | 'left' | 'right' | 'dead';  
        from_down?:  'up' | 'left' | 'right' | 'dead';  
    };  
    colorFilter?: string;// 'none' | 'red' | 'blue' (颜色关卡机制)  
}  
  

### 2.3 进阶操作机制

1. 覆盖与擦除（Dynamic Override）：
    

- 玩家可以将手牌拖拽到已经铺有道路的网格上。
    
- 判定规则：如果 Runner 当前正处于该网格内，禁止覆盖；如果 Runner 尚未到达，新晶体块直接替换旧晶体块，旧晶体块消失。这允许玩家高频修正路线。
    

2. 时空减速（Bullet Time - 核心容错）：
    

- 当玩家长按屏幕下方任意手牌时，全局游戏时钟缩放 Time.timeScale 降至 0.3（变慢3倍）。
    
- 伴随视觉：背景饱和度降低，Runner 出现蓝移特效。松开后恢复 1.0。
    
- 限制：每关有独立的“子弹时间能量条”（如每关限用 3 秒），随时间缓慢恢复。
    

## 3. 关卡与经济系统 (Progression & Economy)

### 3.1 关卡三要素

- 常规块：直道、90度弯道、T字分流道（Runner 遇到 T 字会分裂成两个独立小球，必须两路同时到达终点）。
    
- 机制块（图论/DP核心）：
    

- 单次碎裂块：Runner 离开该网格的瞬间，该网格的 TileConfig 变更为 null（坠落）。
    
- 染色门：Runner 穿过时变为对应颜色；后续必须经过相同颜色的能量网，否则撞碎。
    

- 残局生成器（算法驱动）：游戏关卡不依赖人工堆砌，由后台 DP 算法根据“可用手牌组合”逆向推演，生成有且仅有一条最优解的几何残局。
    

### 3.2 游戏模式

1. 关卡模式（旅途）：固定起点终点，完成特定几何轨迹的折射通关。
    
2. 无尽模式（流光）：没有终点，网格随着 Runner 前进不断向右上方滚动刷新，速度逐渐加快，挑战极限手速。
    

## 4. 视觉与特效规范 (UI & VFX - 零美术实现方案)

本游戏完全不使用第三方外包原画，全部视觉质感由 Cocos Creator 3.8 的材质、着色器和内置组件完成。

### 4.2 特效艺术 (VFX) —— 用代码做动效

- 粒子拖尾 (Tail Effect)：Runner 节点挂载 Cocos 内置 ParticleSystem2D。参数配置：最大粒子数 50，生存时间 0.4s，发射颜色与主题霓虹线一致，结束颜色为全透明。效果：球体滚动时自带晶莹流光尾迹。
    
- 幽灵预览 (Ghost Preview)：当手牌被拖拽至地图上方但未松手时，目标网格上实例化一个透明度为 80 的克隆体，并用 Graphics 动态在方块表面画一个虚线箭头，提示 Runner 经过时的转向趋势。
    

## 5. 微信社交裂变与商业化设计 (Virality & Monetization)

### 5.1 病毒式裂变矩阵

#### 裂变大招一：异步救场链接 —— “帮我续一卡”

- 触发场景：玩家在无尽模式刷新高分或在困难关卡即将通关时失误死亡。
    
- 常规做法：看广告复活。
    
- 裂变优化：点击“呼叫时空导师”，将当前残局数据转化为加密字符串挂载在微信分享链接上。
    
- 群友体验：群友点击小游戏卡片，直接进入完全相同的濒死残局，群友有一次机会在空缺处补齐那个致命的晶体块。成功后，原玩家微信收到服务通知：“@张三 替你接通了浮岛，你已复活并获得无敌光环3秒！”
    

#### 裂变大招二：个性光轨海报 —— “无字情书”

- 触发场景：通关或无尽模式结算。
    
- 表现形式：隐藏所有游戏 UI，只留深色渐变背景和玩家由一轨轨晶体折射出的完整发光路径。路径终点生成一枚旅行邮票印章。
    
- 文案库（根据过关路径随机匹配）：
    

- 直线型路径：“一往无前的路上，谢谢你没有让我拐弯。”
    
- 九曲回肠路径：“兜兜转转，幸好每一步都有你的晶莹承托。”
    

- 目的：极高审美的情感共鸣，驱动女性与年轻用户自发截图分享朋友圈。
    

### 5.2 变现点（极高转化率的广告位）

1. 第四个手牌槽（硬刚需）：玩家初始只有3张候选手牌，极易卡手。看一次激励视频广告，可解锁“第四个备用槽”，持续 24 小时。
    
2. 时空蓄能满载：当关卡内“子弹时间”能量耗尽时，可看广告立即补满。
    
3. 纯代码换色皮肤：看广告解锁“暮色罗兰”或“落日余晖”色彩包。
    

## 6. 技术实现路线图 (Technical Implementation Plan)

### 阶段一：图论状态机验证（1-2天）

让 Codex 辅助完成 MapManager 矩阵定义和 Runner 的定步长网格跳转逻辑，确保方向映射表（DirectionConfig）判定 100% 准确。

### 阶段二：2.5D坐标转换与触控（2天）

实现 UI 层手牌的 TOUCH_START / MOVE / END 拖拽事件。利用屏幕射线与逻辑网格的逆向矩阵公式，实现拖拽时的“幽灵预览”和“精准落子”。

### 阶段三：微信 SDK 接入与打包（2天）

打通微信开放数据域，搭建群排行榜。接入微信分享接口（wx.onShareAppMessage），封装残局数据串，上线测试。
